# 公安智能监控平台（Police Intelligent Monitoring Platform）

> **前后端统一规划文档** — 整合前端全景概览 + 后端架构设计讨论

---
第一：拿来主义与极速交付（UI 驱动）。不从零造轮子，前端底座直接采用国内政企最成熟的 Geeker-Admin 架构进行外科手术式改造。配合开源界的成熟安防组件，确保在 M1 阶段（前四周）以最快速度让系统“跑起来、能看流”。

第二：前置规避核心风险（协议解耦）。公安安防项目的生死线在于“视频流性能”和“告警实时性”。我的想法是必须在开工第一周严格界定推流协议（强烈建议 WebRTC/FLV）和告警协议（SSE），将前端渲染压力与后端流媒体转码彻底解耦，保障 16 宫格下的性能极限。

第三：为大模型留足后路（业务闭环）。考虑到 M3 阶段及未来的“公安智库”需求，我在前期的架构设计和状态机定义时，就将告警数据、视频截图与业务链条打通，为后续通过 RAG（检索增强生成）或 Agent 接入大模型预留标准化 API 和数据飞轮的入口。
## 一、项目定位

基于 **Geeker-Admin**（Vue 3 + TypeScript + Vite 5 + Element Plus）构建的政企安防监控后台，后端使用 **MySQL + Python**，核心能力覆盖：实时视频监控、多场景告警闭环、工作流编排。

### 通信协议选型理由

| 场景               | 协议选型              | 架构考量（为什么这么用）                                                                 |
| ------------------ | --------------------- | ---------------------------------------------------------------------------------------- |
| **常规 CRUD 业务** | **HTTP (Axios)**      | 绝对主力。用于点位管理、系统配置等非实时交互。通过 Vite/Nginx 优化支持 HTTP/2 压缩。     |
| **监控画面传输**   | **WS-FLV / WebRTC**   | 视频流专属。Jessibuca 播放器底层支持，针对安防低延迟做了硬件加速和缓存优化。             |
| **实时告警推送**   | **SSE (流式更新)**    | **轻量级选型**。告警属于服务器向前端的单向“推流”，SSE 比 WebSocket 更省资源且自带重连。 |
| **未来云台控制**   | **WebSocket (预留)** | **双向高频互动**。针对未来 PTZ（摇头、缩放）等需要毫秒级双向反馈的场景预留接口。         |

> **架构师解读**：
> - **为什么不用轮询？** 轮询像“对讲机”，每秒询问浪费带宽和服务器 CPU。
> - **为什么告警用 SSE 而非 WebSocket？** 告警是典型的单向通知，SSE 协议开销小，像“单向电话”，且完美适配 Nginx 代理和多路复用。
> - **功耗考量**：SSE/WebSocket 比起轮询更省电费、省流量，适合政企长期挂机运行的环境。

> **注意**：Nginx 反向代理需配置 `proxy_buffering off`，否则 SSE 消息会被缓存，导致实时性差。

## 三、里程碑规划

### M1（第 1-4 周）：系统基础 + 视频展示

| 验收项                    | 前端状态            | 后端接口                                        |
| ------------------------- | ------------------- | ----------------------------------------------- |
| 登录/退出/会话管理        | ✅ 框架自带         | —                                               |
| 角色权限控制              | ✅ 框架自带         | —                                               |
| 点位（视频流）列表 + 筛选 | ✅ `views/point/`   | `GET /api/streams`                              |
| 点位在线/离线状态刷新     | ✅ 状态标签映射     | `GET /api/streams`（status 字段）               |
| 新增 / 编辑 / 删除点位    | ✅ 弹窗表单         | `POST` / `PATCH` / `DELETE /api/streams/{id}`   |
| 手动立即检查              | ✅ 按钮             | `POST /api/streams/{id}/check`                  |
| 单路视频播放              | ✅ `views/monitor/` | `GET /api/streams/{id}/preview`（返回播放地址） |
| 多宫格视频（4/9/16）      | ✅ CSS Grid 布局    | 子码流 / 主码流切换                             |
| 视频切换 / 全屏 / 静音    | ✅ 播放器封装       | —                                               |
| 时间轴回放                | 🔲                  | 需后端提供录像时间片段列表                      |

#### 后端视频流状态机（已确认）

```
新增流 → checking → available ⇄ unavailable
                           ↘              ↙
                          disabled（人工禁用，跳过所有任务）
```

#### 后端核心数据模型 `video_stream` 主要字段

`id` / `name` / `source_url`（加密）/ `protocol` / `status` / `last_check_time` / `last_error` / `consecutive_failures`
