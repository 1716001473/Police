# 智慧视频监控系统

- 记录日期：2026-03-03

## 0. 基础设施与存储设计

### 0.1 结构化存储选型

系统使用 MySQL 作为结构化数据存储，承载业务主数据、状态数据和可审计数据。

### 0.2 MySQL 承载的数据范围

1. 视频流管理数据：流配置、状态、巡检结果、错误摘要。
2. 场景管理数据：场景配置、规则参数、场景与视频流的绑定关系。
3. 报警系统数据：报警事件、报警级别、处理状态、通知记录。
4. 通用审计数据：创建人、更新时间、操作日志、软删除标记。

### 0.3 数据分层原则

1. 全局层（本章节）：定义 MySQL 作用范围和数据分类。
2. 模块层（各业务章节）：定义本模块具体表结构和关键字段。
3. 扩展层（后续）：按性能和规模再引入缓存、时序库或对象存储。

## 1. 视频流管理模块

### 1.1 目标与范围

目标：统一管理所有摄像头视频流，提供稳定接入、状态可视化、可回放查看和巡检能力，为后续场景检测与报警提供可靠输入。

本模块负责：

1. 视频流配置管理（增删改查、启用/禁用）。
2. 视频流可用性状态管理（自动巡检 + 手动检查）。
3. 视频流查看能力（点击可预览）。
4. 与检测任务联动（可用时运行，不可用时暂停）。

本模块暂不负责：

1. 具体算法推理逻辑（YOLO/大模型本体）。
2. 报警消息发送策略本体（由报警系统负责）。

### 1.2 核心需求（确认版）

1. 添加视频流。
2. 列出所有视频流并展示当前状态。
3. 点击视频流可查看。
4. 系统定期检查视频流是否可用。
5. 支持编辑、删除、启用、禁用。
6. 支持手动“立即检查一次”。

### 1.3 状态机设计

建议使用以下状态：

1. `checking`：系统正在做连通性检查。
2. `available`：流可正常拉取。
3. `unavailable`：流当前不可用。
4. `disabled`：人工禁用，系统跳过巡检和检测任务。

状态流转建议：

1. 新增流后先进入 `checking`，检查成功转 `available`，失败转 `unavailable`。
2. 定时巡检期间，`available/unavailable` 可相互切换。
3. 任意状态都可被人工切换到 `disabled`。
4. `disabled` 被启用后回到 `checking`，再判定可用性。

“禁用流”定义：保留配置和历史，但系统不拉流、不检测、不巡检、不触发报警。

### 1.4 数据模型建议

主表：`video_stream`

1. `id`：主键。
2. `name`：流名称（如“北门-1号摄像头”）。
3. `source_url`：原始流地址（RTSP/HTTP 等，敏感信息需保护）。
4. `protocol`：协议类型（`rtsp`/`http`/`rtmp`）。
5. `status`：当前状态（`checking/available/unavailable/disabled`）。
6. `enabled`：是否启用（可与 `status` 合并实现，按实现习惯选择）。
7. `last_check_time`：最近巡检时间。
8. `last_ok_time`：最近成功拉流时间。
9. `last_error`：最近错误摘要（超时/鉴权失败/连接拒绝等）。
10. `consecutive_failures`：连续失败次数。
11. `consecutive_successes`：连续成功次数（用于恢复判定）。
12. `created_at`、`updated_at`：审计字段。

建议日志表：`video_stream_health_log`

1. `stream_id`、`check_time`、`result`、`latency_ms`、`error_detail`。
2. 用于排障、报表、可用率统计。

### 1.5 接口草案（MVP）

1. `POST /api/streams`：新增视频流。
2. `GET /api/streams`：流列表（支持按状态过滤）。
3. `GET /api/streams/{id}`：流详情。
4. `PATCH /api/streams/{id}`：修改名称、地址等配置。
5. `POST /api/streams/{id}/enable`：启用流。
6. `POST /api/streams/{id}/disable`：禁用流。
7. `DELETE /api/streams/{id}`：删除流（建议软删除）。
8. `POST /api/streams/{id}/check`：立即检查一次。
9. `GET /api/streams/{id}/preview`：获取预览播放地址（如 WebRTC/HLS 地址）。

### 1.6 可用性巡检策略

1. 巡检周期：默认每 30 秒或 60 秒一次（可配置）。
2. 单次超时：默认 3-5 秒（网络环境差可放宽）。
3. 降级阈值：连续失败 N 次（如 3 次）标记 `unavailable`。
4. 恢复阈值：连续成功 M 次（如 2 次）恢复 `available`。
5. 巡检并发：限制并发数，避免短时压垮网络或设备。
6. 日志记录：每次巡检写入健康日志，保留错误原因。

### 1.7 视频查看方案

1. 管理端点击“查看”后，后端返回安全播放地址而非原始源地址。
2. 若前端是浏览器，不直接播 RTSP，建议转为 WebRTC/HLS。
3. 播放失败时展示最近错误信息和“立即检查”按钮，便于排障。

### 1.8 安全与权限

1. 流地址中的账号密码不得明文展示。
2. 敏感配置建议加密存储（至少数据库层脱敏 + 应用层访问控制）。
3. 接口按角色控制：查看、编辑、禁用、删除应区分权限。
4. 日志中避免输出完整密钥/完整 URL。

### 1.9 与检测系统联动

1. 状态为 `available` 且 `enabled` 时，允许创建或运行检测任务。
2. 状态切为 `unavailable` 时，暂停任务并记录原因。
3. 状态恢复 `available` 后，自动重试拉起检测任务。
4. `disabled` 状态下不自动拉起任何任务。

### 1.10 迭代建议

1. 第一阶段（MVP）：增删改查 + 状态展示 + 定时巡检 + 手动检查 + 基础预览。
2. 第二阶段：可用率报表、失败原因统计、批量操作（批量启用/禁用）。
3. 第三阶段：多节点探测、区域容灾、智能诊断（自动定位故障点）。
